# Glossary of Meta Terms

**TL;DR** — Essential terminology and definitions for the AI-Native Meta Documentation System.

## Core Meta Terms

### Amendment
**Definition** — A formal change to meta system structure or principles requiring governance approval
**Context** — Used when modifying the fundamental structure or rules of the MetaDocs system
**Usage** — Applied to significant changes that affect how the system operates

### Chain Prompt
**Definition** — A structured sequence of prompts guiding AI through multi-step documentation tasks
**Context** — Part of the original operational system for complex documentation workflows
**Usage** — Used for breaking down complex documentation generation into manageable steps

### Context Slice
**Definition** — A specific portion of documentation that can be independently updated or referenced
**Context** — Enables modular documentation that can be updated without affecting other sections
**Usage** — Applied to documentation sections that can be maintained independently

### Layer
**Definition** — Numbered directory (0-5) with specific semantic purpose
**Context** — Original architectural concept for organizing meta documentation
**Usage** — Referenced in historical context of the system evolution

### Template
**Definition** — Reusable document structure with placeholders
**Context** — Core component of the MetaDocs system for consistent documentation
**Usage** — Applied to all document blueprints (PRD, Design, Implementation Plan)

### Docs Layer
**Definition** — The canonical `/docs` folder structure for project documentation
**Context** — The standardized structure that gets generated for each project
**Usage** — The target structure that MetaDocs creates in new projects

### Framework
**Definition** — A template or blueprint for creating specific types of documentation
**Context** — Part of the original 1-Frameworks layer concept
**Usage** — Referenced in historical context of the system

### Meta Layer
**Definition** — The `/meta` folder containing documentation about the documentation system itself
**Context** — The self-referential layer that defines how the system works
**Usage** — Contains handbook, glossary, and templates that govern all documentation

### Self-Governance
**Definition** — The meta system's ability to maintain and evolve itself through automated processes
**Context** — Core principle that the system can adapt and improve itself
**Usage** — Applied to the system's ability to update its own standards and practices

### Vibe Coding
**Definition** — Development approach where humans code through natural-language prompts with AI assistance
**Context** — The collaborative development style that MetaDocs supports
**Usage** — Describes the human-AI collaboration model the system enables

### MCP
**Definition** — Model Context Protocol for providing structured context to AI agents
**Context** — Technical protocol for enhancing AI capabilities with structured data
**Usage** — Referenced in context of AI agent communication and context management

### Feature Pack
**Definition** — A collection of related documentation for implementing a specific feature
**Context** — Part of the original operational system for feature development
**Usage** — Applied to comprehensive feature documentation sets

### Meta-Changelog
**Definition** — Version history tracking evolution of the meta documentation system
**Context** — Historical tracking of changes to the MetaDocs system itself
**Usage** — Used for maintaining version history of the meta system

### Override Declaration
**Definition** — Formal documentation of project-specific customizations to meta system rules
**Context** — Part of the original Adaptations layer concept
**Usage** — Applied when projects need to customize standard MetaDocs practices

### Local Government
**Definition** — The Adaptations layer's authority to define project-specific rules and exceptions
**Context** — Part of the original 4-Adaptations layer concept
**Usage** — Referenced in historical context of project customization

### Speech-to-Text Awareness
**Definition** — AI consideration for potential misinterpretations in voice-based communication, especially proper nouns and technical terms
**Context** — Critical for interpreting user input that may contain transcription errors
**Usage** — Applied when AI processes voice-to-text input from users

## System Architecture Terms

### MetaDocs Template System
**Definition** — Universal AI documentation system that generates clean, concise project documentation automatically
**Context** — The core system that provides structure and standards for all project documentation
**Usage** — Copy entire /docs structure into new projects to bootstrap documentation

### AI Documentation Standards
**Definition** — Core principles for AI-generated documentation: assume intelligence, be concise
**Context** — Ensures documentation doesn't explain obvious concepts that AI already knows
**Usage** — Applied to all generated project documentation

### Development Workflow
**Definition** — Essential development practices: build verification, test execution, commit & push
**Context** — Standardized process for all development work
**Usage** — Each implementation step must include build/test/commit/push workflow

### Template Rules
**Definition** — Standards for using templates: TL;DR, AI standards, plain markdown, under 300 lines
**Context** — Ensures consistency across all generated documentation
**Usage** — Applied to all template usage and document generation

### Project Documentation
**Definition** — Documentation generated using MetaDocs templates for specific projects
**Context** — The actual documentation that gets created for each project
**Usage** — Becomes the primary reference after MetaDocs steps into background
